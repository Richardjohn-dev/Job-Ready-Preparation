# 30-Day Operating Plan

Current as of 2026-06-02.

Purpose: run the first practical career-readiness month without turning preparation into dashboard work. This plan prepares Richard for issue #12 diagnostic practice, issue #13 market-test execution, issue #10 RefineBridge discovery, and issue #11 dashboard decision.

Default start: Day 1 is the first day Richard actively starts the plan. If started immediately, use 2026-06-03 to 2026-07-02.

## Non-Negotiables

- Do not mark any application or outreach as sent during issue #9.
- Do not close or mark GitHub issues complete from this plan.
- Do not build a Notion dashboard, local dashboard, or tracking app during the first two weeks.
- Count only output-producing work: artifacts, answer drafts, drills, logged feedback, applications, outreach, or tracker updates.
- Keep claim boundaries from `cv-draft.md`, `linkedin-profile-draft.md`, `codex-interviewer-workflow.md`, and case-study files.
- Re-check every job ad manually before issue #13 execution because role links may be stale.

## Default Time Split

Use this split unless a real interview, recruiter call, or application response changes the week.

| Area | Default share | Examples |
| --- | ---: | --- |
| Hiring funnel and market test | 35% | role re-checks, CV tailoring, outreach variants, `application-tracker.md` prep |
| Interview delivery and story practice | 25% | 30-second intro, RDrive walkthrough, behavioral story rewrites, issue #12 diagnostic |
| Practical technical drills | 20% | .NET API, EF query, webhook idempotency, failed sync debugging |
| Feedback and weak-spot updates | 10% | reviewer asks, feedback notes, score changes, weekly review |
| RefineBridge discovery | 10% | customer hypotheses, discovery questions, target categories, issue #10 prep |

Dashboard/Notion work allocation before issue #11: `0%`, except listing friction points in the weekly review.

## Daily Minimum Actions

On any active workday, complete at least two of these:

- Produce one answer, drill note, tracker update, role note, or feedback note.
- Re-check one candidate role link for live status, region, timezone, seniority, stack, and application path.
- Do one 15-30 minute spoken or written practice rep tied to a weak spot.
- Complete one practical drill from `practical-technical-drills.md`, `system-design-practice-pack.md`, or `reliability-security-interview-pack.md`.
- Update one affected artifact only when evidence supports it.

Do not count passive reading unless it produces a logged output.

## Weekly Minimum Actions

Each week should produce:

- 1 weekly review using `weekly-review-template.md`.
- 1-2 weak-spot or practice-log updates backed by a real output.
- 1 CV, LinkedIn, story, or outreach improvement if evidence justifies it.
- 1 feedback action or prepared feedback ask.
- 2-4 technical/story practice outputs.
- Market-test progress appropriate to the week, without logging sent applications until issue #13.

## Dashboard / Notion Protection

Issue #11 is a decision gate, not a build task.

Before issue #11, allowed dashboard work is limited to:

- writing friction points in weekly reviews
- noting repeated fields that are painful to maintain
- identifying what Markdown fails to show clearly

Blocked before issue #11:

- Notion database design
- local dashboard implementation
- automation scripts for tracking
- elaborate charts
- new productivity tooling

Build only if issue #11 finds that:

- the manual workflow has been used for at least 2 weeks or through one diagnostic sprint plus one market-feedback loop
- pain points are specific
- expected dashboard benefit beats maintenance cost
- a separate PRD or issue set is created

Default decision if evidence is weak: keep Markdown and revisit after the next market-feedback loop.
